# Zig Tutorials: Notes for AI Agents

There's one self-contained `.zig` file per tutorial under `src`. The filenames
match the other ports (`send`, `receive`, `new_task`, `worker`, the four
`emit_log*` and `receive_logs*` pairs, `rpc_server`, `rpc_client`,
`publisher_confirms`). When the wording or behavior is ambiguous, look at the
Ruby port: that's the one we treat as canonical.

## Zig Version

Builds use Zig 0.16+ against [`bunny-zig`](https://github.com/michaelklishin/bunny-zig),
pulled in via `build.zig.zon`.


## Build System

 * `zig build` compiles every binary
 * `zig build run-<name> -- args...` runs a single tutorial


## Zig Code Guidelines

 * Stick to `std.heap.smp_allocator` at runtime. The debug allocator complains
   about allocations that bunny-zig legitimately holds for the lifetime of the
   connection (server-sent field tables), and the noise at exit isn't worth it
 * Print via `std.Io.File.stdout().writerStreaming(io, &buf)` and call
   `try out.flush()` after each line, so consumer parts of the tutorials actually show output
   live. `std.debug.print` goes to stderr, which makes Zig diverge from
   other tutorials (e.g. Ruby, Swift) when output is piped
 * These are self-contained learning materials, so keep them short and simple to the point of being dumb.
   Don't introduce abstractions. Don't try to eliminate duplication


## `bunny-zig` Guidelines

 * For server-named exclusive queues, use `Channel.temporaryQueue`. For
   client-named ones, use `quorumQueue("name")`, `durableQueue("name")`, or
   `queueDeclare("name", opts)`. They all return a `Queue` handle. Its
   `deinit` is a no-op for borrowed names and frees the duped name for
   server-named ones, so `_ = try ch.quorumQueue("hello")` is already
   leak-free
 * Exchange declarations (`declareFanoutExchange`, `declareDirectExchange`,
   `declareTopicExchange`, `declareHeadersExchange`, `exchangeDeclare`)
   return an `Exchange` handle. In producers, capture it and call
   `exchange.publish(body, routing_key, props)` — shorter than
   `ch.publish(body, .{ .exchange = ..., .routing_key = ... })`. In
   consumers, where the exchange is only declared so a binding has a target,
   just `_ =` the result
 * Cleanup: `defer ch.close();` (void, swallows errors) for channels, and
   `defer conn.deinit();` for connections. `closeChannel()` is an alternative
   that returns an AMQP error code
 * Consumer loops: `Delivery`, `BasicGetResult`, and `ReturnedMessage` all take
   themselves by reference inside `deinit`, but their signatures pass by value.
   That means
   `while (try ch.recvDelivery()) |delivery| { defer delivery.deinit(allocator); ... }`
   works straight off the capture, without a `var msg = raw;`-like rebinding
 * Acks: `Delivery.ack` is the manual-ack one-liner; siblings are
   `delivery.nack`, `Delivery.nackRequeue`, `Delivery.reject`,
   `Delivery.rejectRequeue`. `Delivery` carries the
   originating channel internally, so RPC servers can also call
   `delivery.respond(body)` without passing in the `ch`. `Channel.ack(tag)`,
   `Channel.nack(tag)` and other acknowledgement API methods pass the delivery tag on
 * `recvDelivery` blocks. For consumer tutorials, this is acceptable: they use a single-threaded loop and let `Ctrl+C`
   invoked by the user tear down the process
 * For per-consumer QoS, prefer the `ch.prefetch(n)` shorthand. Steer clear
   of `basicQos(n, true)`: the `global_qos` deprecated feature is denied by
   default starting with RabbitMQ 4.3.0, so a stock broker will reject the
   channel-wide variant
 * Publisher confirms: `confirmSelectWithOptions(.{ .tracking = true })`
   enables "per-message" publisher confrim with automatic tracking.
   Regular `Channel.confirmSelect` (with tracking disabled) plus an explicit `waitForConfirms` at the end
   covers the batch and asynchronous modes. Note that with tracking on,
   `Channel.publish` already awaits each confirm, so `outstanding_limit`
   only matters for `publishAsync` and `publishBatch`


## Cross-port Testing

Queue and exchange names line up with the other ports (`hello`, `task_queue`,
`rpc_queue`, `logs`, `direct_logs`, `topic_logs`), so a Zig producer or
consumer should interoperate with any Ruby/Swift/etc. counterpart.

## Git Instructions

 * Never add yourself to the list of commit co-authors
 * Never mention yourself in commit messages in any way (no "Generated by", no AI tool links, etc)

## Style Guide

 * Add comments only when the intent isn't obvious from the code, and keep
   them short
 * Never add full stops to Markdown list items


## After Completing a Task

### Iterative Reviews

After completing a task, perform up to twenty iterative reviews of your changes.
In every iteration, look for meaningful improvements that were missed, for gaps in test coverage,
and for deviations from the instructions in this file.
